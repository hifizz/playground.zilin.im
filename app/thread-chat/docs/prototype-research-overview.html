Thread Chat：分支对话的交互方案调研

把线性聊天升级为「可导航的思考树」——六个原型的对照与取舍
面向 · AI 学习产品 v1（Chatbot 形态） ｜ 状态 · 交互原型阶段 ｜ 产出 · 6 个可交互 HTML 原型 + 本对照文档

01 Motivation · 我们为什么要做这个

起点是一个具体的产品设想，也是一个关于「人怎么思考」的观察。

今天的 Chatbot 都是线性的：你和 AI 的对话是一条从上到下的直线。但真实的思考不是线性的。当 AI 给出一段回答，里面往往埋着好几个你想追问的点——某个术语、某个论断、某个例子。如果你顺着其中一个追下去，主线的思路就被打断了；如果你把好几个追问都堆在同一条线里，最后自己都理不清哪个回答对应哪个问题。

Thread Chat 想解决的就是这件事：让用户能划选 AI 回复里的任意一段文字，就地开启一个分支对话。这个分支继承从对话开头到划选处的全部上下文，背后是一套完整的 Chatbot。用户在分支里把这个点聊透，再回到主线继续——针对同一个回答提出不同的问题、分别聊清楚、再收束回主线。

一句话目标 把一条线性的对话，变成一棵可导航的思考树：主干是主线，枝条是分支，节点之间可回溯、可对照。这契合人「发散—收敛」的思考节奏，而不是强迫思考迁就聊天框的线性。

02 核心交互与目标状态

四个动作构成完整闭环。

  • 划选：在任意 AI 回复里选中一段文字，浮出一个迷你对话框。
  • 开分支：确认后开启一个新的分支对话，被划选的文字成为该分支的「讨论焦点」。
  • 继承上下文：分支自动继承从对话开头到分叉点的全部消息（分叉之后主线与分支互不干扰）。
  • 回溯与对照：分支聊完可以回主线；分支里的结论可以「带回主线」；被划选过的文字留下脚注式标记，随时能重新打开对应分支。

再往上，分支本身可以再开分支（嵌套），于是整个会话长成一棵树。产品要解决的核心难题，就是怎么在有限的屏幕上，让用户舒适地浏览和导航这棵树——这正是六个原型在各自回答的问题。

03 关键约束 · 一切方案都绕不开的硬骨头

这些约束是在原型迭代中逐步被逼出来的，它们决定了方案的边界。

约束一：横向空间是有限的，而分栏假设它无限

把每个分支做成一列并排展示，最直观。但屏幕宽度有限，分支一多，列就装不下。几乎所有分栏方案的麻烦，根子都在这里。

约束二：人的对照带宽只有 3–4 列

就算屏幕够宽，同时打开 7 列对话本身就是反人类的——没有人能同时对照那么多。所以「把 7 列排整齐」是伪命题，真正的解法是「根本不让它变成 7 列」。

约束三：分支树是二维的，屏幕列是一维的

一个典型的错乱场景：在第 4 列划选 3 句话开了第 5/6/7 列（互为兄弟），又从第 6 列往下开子分支。这个子分支该插在哪？插在 6 和 7 之间会把 7 挤走、打乱用户的空间记忆；追加到最右边又会让它和父亲（第 6 列）被第 7 列隔开。一维的列序，天然装不下二维的树。

约束四：Artifact 会和分栏抢横向空间

如果每个分支还能在右侧展示 artifact（代码、文档、图表），它和分栏争的是同一个横向维度，会让空间问题雪上加霜。

约束五：划选锚点在富文本下会失效

用户看到的「第 100 个字符」和数据库里存的未必一致——Markdown 渲染（粗体、代码块、链接）会让 offset 错位。这是工程实现层面必须专门处理的坑。

约束六：移动端不成立

分栏在手机上根本铺不开。任何最终方案都要有移动端的降级形态（单列钻取 / 全屏切换）。

收敛出的核心原则 不要让横向空间无限扩张。给它一个硬上限，用「坞 / 地图 / 焦点」三件套管理超出上限的部分——这条原则同时解决了约束一、二、三，是后面几个原型的共同底座。

04 功能子集清单

在 v1 / v2 的讨论里，我们把功能按优先级分成了四层。原型阶段验证的是交互骨架（必做层），进阶层留待后续。

必做 · v1 骨架

  • 划选开分支 + 迷你对话框
  • 分支继承分叉点前的全部上下文
  • 分支自动生成标题（不打断思考流）
  • 主线消息上标记「此处有分支」（脚注标记）
  • 分支列表 / 树的基本展示与导航

进阶 · 有余力再做

  • 分支结论「带回主线」（手动按钮版）
  • 跨分支引用（@ 弱引用）
  • 分支组织与检索（归档 / 搜索 / 分组）
  • 跨会话的持久记忆（Memory Bank 基础版）

观望 · 有反馈再定

  • 嵌套分支的完整 UI 引导（数据层已支持）
  • 分支的知识图谱 / 全景视图
  • Artifact 与分支的深度融合（分支作为记忆产生器）

暂不做 · 违背产品初衷或属 v3

  • 分支自动合并回主线（违反「不打扰主线」）
  • AI 主动建议开分支（干扰思考流）
  • 跨用户共享分支（涉及权限协作，属 v3）

05 两个交叉议题的结论

这两个问题横跨所有布局方案，是原型迭代中专门讨论并落地的。

议题 A：Artifact 与分栏抢位置 → 独立舞台

结论是双区布局：分栏区在左（横向可扩），Artifact 有一个右侧固定的独立舞台，全局唯一、用标签页管理多个 artifact、标签圆点用深度色标注来源分支。宽屏常驻，窄屏降级为 Drawer 盖上来。关键决策是「全局单一舞台」而非每列各开——否则 artifact 会二次放大分栏的混乱。（已实现于原型 ③④⑤）

议题 B：分栏顺序错乱 → 约束消灭混乱

不用算法去「整理」混乱，而是用约束「消灭」混乱：限制同时展开的列数（3–4），超出的分支不消失，而是收进分支坞 / 折叠成细条 / 被替换（配面包屑回溯）。父子关系与列序始终稳定。（已实现于原型 ③⑤⑥）

06 六个原型的结构示意与说明

六个原型共用同一套视觉语言（手稿批注风 + 脚注式分支标记 + 深度分色），差异全在布局与导航机制。下面每张示意图标出了各自的「签名机制」。

① 基础分栏（全平铺）

thread-chat-demo.html
所有打开的分支并排成列，一眼看全貌。开分支即往右加一列，横向可滚动。最朴素、最直观的形态。
优点
  • 全貌可见，对照最直接
  • 心智模型最简单
缺点
  • 分支多了横向爆炸
  • 开新分支会挤动已有列
适合：验证核心交互；分支很少的轻量场景。作为后续所有版本的基线。

② 侧栏 + 面包屑（单面板钻取）

thread-chat-demo-sidebar.html
主线常驻左侧，右侧只有一个分支面板，一次看一个分支。往深钻时面板「替换」内容，顶部面包屑记录整条路径，点面包屑回退。另有分支地图看全貌。
优点
  • 永远只占一列宽，省屏幕
  • 天然适配移动端
  • 专注、干扰小
缺点
  • 看不到全貌（靠地图补）
  • 切换兄弟分支要多点几下
适合：移动端 / 窄屏；顺着一条线深挖的专注场景。

③ 横条管理版（坞 + 舞台）

thread-chat-demo-managed.html
「三件套」首次集齐：主线 + 最多 2 分支列，超出的自动退回顶部横条（分支坞）；右侧独立 Artifact 舞台；外加分支地图、pin、关闭。
优点
  • 横向永不超 3 栏
  • Artifact 不抢位置
  • 功能最完整
缺点
  • 被挤的列「消失」到横条，空间连续性弱
  • 横条层级表达浅
适合：功能完整度优先、桌面为主的重度工具。

④ 左侧竖栏树导航

thread-chat-demo-rail.html
分支坞从顶部横条改为左侧竖直树：真正的树形缩进、子树可展开/折叠、整条栏可收成图标窄条。其余三件套与 ③ 一致。
优点
  • 能表达很深的树
  • 结构感最强
  • 可折叠省空间
缺点
  • 吃一条水平宽度
  • 浅树时略重
适合：分支多且深、需要强结构感（如层层深挖一个知识点）。

⑤ 限栏 + 细条折叠

thread-chat-demo-capped.html
最多展开 3 栏。开第 4 个时，最久没碰的那栏在原地缩成一根竖直细条（不消失、不换位），点细条就地展开。顶部横条 + 舞台 + pin + 地图齐全。
优点
  • 列不瞬移，空间连续性最好
  • 细条一眼可见可点
  • 约束彻底消灭错乱
缺点
  • 细条累积会变长（靠滚动）
适合：平衡「对照爽感」与「不混乱」，综合最接近可直接落地。

⑥ 自适应列数 + 面包屑替换

thread-chat-demo-responsive.html
列数按屏幕宽度自动定（约每 430px 一列，2–4 之间：MacBook 得 3、4K 得 4，可手动切）。列满后再往深钻，替换来源列，该列面包屑累积整条路径，点面包屑就地回退。
优点
  • 贴合屏幕，横向永远整齐
  • 面包屑承载无限深度
  • 无细条杂物，最干净
缺点
  • 替换是破坏性的——被替换分支退出视野，需靠面包屑/地图召回
适合：「层层深入理解一个概念」的线性深挖，最贴合 AI 学习产品。

08 下一步交互调研问题

原型阶段验证了「桌面端的树导航」，但要走向产品，还有几个未决问题需要专门调研。

  1. 移动端形态：分栏在手机上不成立。是全量走「侧栏面包屑（②）」，还是桌面分栏、移动端自动降级为单列钻取？断点如何切换？这是决定产品能否移动优先的关键。
  2. 破坏性替换 vs 空间连续性的最终取舍：⑥ 的「替换」最干净但会丢失被替换分支的可见性；⑤ 的「细条」保留可见性但会累积。需要用真实使用数据（分支深度分布、兄弟分支对照频率）来定夺，而不是凭感觉。
  3. 划选锚点在 Markdown 富文本下的鲁棒定位：offset 会因渲染错位。需调研「存文本片段 + 前后上下文做定位」等方案，这是工程实现的第一个硬骨头。
  4. 分支的持久化与跨会话记忆：把分支结论沉淀为「跨会话的知识库」——分支作为记忆的天然产生器。数据模型（树 vs DAG）、写入/读取时机需要设计。
  5. 首次引导与空态：「划选开分支」不是用户的既有习惯，需要设计引导让用户形成这个动作。这决定了功能的实际使用率。
  6. Artifact 与分支的融合深度：舞台已解决「抢空间」，但更进一步——分支能否自动生成 artifact、artifact 能否反向锚定回分支？这是「观望层」里最有潜力的一条。
